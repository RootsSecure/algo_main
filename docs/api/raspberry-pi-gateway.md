# Raspberry Pi and Edge Device Gateway Connection

## Purpose

This gateway is a dedicated hardware connection lane for a Raspberry Pi, a normal PC, or another small edge device running a camera or sensor agent. It is separate from owner or operator login and is intended for device-to-platform connectivity only.

## What can use this gateway

- Raspberry Pi with a Pi Camera module
- Windows PC with a webcam or USB camera
- Linux PC with a USB or IP camera
- Mini PC or edge box near the property

The backend does not require Raspberry Pi-specific hardware. The client just needs to follow the same secure connection flow.

## Security model

The connection uses two token stages:

1. Provisioning token
- Generated by an authenticated owner or ops admin for a specific device
- Intended for secure device bootstrap or token rotation
- Long-lived compared with the session token, but still revocable by issuing a new provisioning token

2. Session token
- Issued only after the edge device successfully connects with the provisioning token
- Short-lived compared with the provisioning token
- Bound to the device id and hardware id
- Used for ongoing heartbeats and event ingestion

This separation keeps hardware traffic out of the normal user-auth flow and gives you a clear credential-rotation boundary.

## Endpoints

### 1. Provision device gateway credentials
`POST /api/v1/gateway/devices/{device_id}/provision`

Auth: owner or ops bearer token

Request:
```json
{
  "hardware_id": "edge-plot-east-001"
}
```

### 2. Edge device connect
`POST /api/v1/gateway/raspberry-pi/connect`

Auth: provisioning bearer token

Even though the route name includes `raspberry-pi`, the same route is valid for a normal PC-based edge agent.

Request:
```json
{
  "hardware_id": "edge-plot-east-001",
  "network_status": "online",
  "power_status": "healthy",
  "battery_level": 97,
  "ip_address": "10.0.0.24",
  "client_version": "sentinel-edge-0.1.0",
  "camera_model": "Pi Camera Module 3 or USB Webcam",
  "metadata_json": {
    "stream_profile": "1080p-low-latency"
  }
}
```

### 3. Edge device heartbeat
`POST /api/v1/gateway/raspberry-pi/devices/{device_id}/heartbeat`

Auth: session bearer token

Use this for regular health and connectivity updates.

### 4. Edge device event ingestion
`POST /api/v1/gateway/raspberry-pi/devices/{device_id}/events`

Auth: session bearer token

Use this for tamper, motion, offline, gate-breach, or manually classified events from the device-side software.

## Recommended deployment pattern

- Keep the provisioning token off the day-to-day event path
- Use it only for bootstrapping or session renewal
- Store the session token in memory when possible and refresh by reconnecting
- Protect both tokens using HTTPS transport and local file permissions on the device
- Bind each physical device to a stable `hardware_id`

## Suggested hardware id format

Use a deterministic value such as:
- `edge-jaipur-plot-01`
- `edge-boundary-east-001`
- `edge-gate-cam-01`

## Suggested client responsibilities

- Read camera or sensor events locally
- Normalize events to the API schema
- Maintain periodic heartbeat updates
- Reconnect when the session token expires
- Queue events locally if mobile connectivity drops temporarily
